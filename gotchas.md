> Continuously updated — each real failure gets documented immediately.

## Collection Layer (V2.0.0 New)

### ⚠️ MiniMax Feishu `_signature` cannot be curl'd directly
- **Symptom**: Feishu ATS `/api/v1/search/job/posts` requires `_signature` parameter, dynamically generated by browser JS
- **Root cause**: Feishu ATS is a pure CSR app, `_signature` algorithm is obfuscated in JS bundle, not reversible
- **Fix**: Must use `playwright` to let the browser execute JS, then intercept network requests for the signed API response
- **Verified**: 2026-04-10

### ⚠️ agent-browser doesn't support CONNECT proxy (Feishu ATS can't use agent-browser)
- **Symptom**: agent-browser (built-in Playwright) opening `vrfi1sk8a0.jobs.feishu.cn` reports `net::ERR_TUNNEL_CONNECTION_FAILED`
- **Root cause**: agent-browser's Playwright instance doesn't support CONNECT tunnel proxies
- **Fix**: Use system-level `playwright` Python library (`from playwright.sync_api import sync_playwright`), pass `proxy={"server": "..."}` in `launch()`
- **Verified**: 2026-04-10

### ⚠️ Camoufox + proxy also doesn't work (missing libgtk-3)
- **Symptom**: `camoufox.sync_api.Camoufox(proxy=...)` startup fails with `libgtk-3.so.0: cannot open shared object file`
- **Root cause**: Container lacks GTK3 libraries
- **Temp fix**: `apt install libgtk-3-0` (not persisted)
- **Current choice**: Use system playwright instead, no GTK dependency
- **Verified**: 2026-04-10

### ⚠️ `dedup_jobs()` depends on `id` field, but MiniMax Feishu jobs have no `id`
- **Symptom**: MiniMax collection results have no `id` field (Feishu ATS doesn't expose `job_id`), KeyError during dedup
- **Fix**: `dedup_jobs()` now prioritizes `id`, falls back to `url`, then to `{company}:{title}:{location}`
- **Verified**: 2026-04-10

### ⚠️ `daily_diff.py` `location` vs `city` field incompatibility
- **Symptom**: MiniMax Feishu jobs use `city` field, other companies use `location`; diff script direct `j['location']` raises KeyError
- **Fix**: `j.get('location') or j.get('city', '')`
- **Verified**: 2026-04-10

### ⚠️ Proxy for Feishu ATS only works via CONNECT tunnel, not HTTP proxy
- **Symptom**: HTTPS domains (vrfi1sk8a0.jobs.feishu.cn) can only be accessed via CONNECT method tunnel
- **Fix**: playwright `proxy={"server": "http://YOUR_PROXY:PORT"}` works — playwright internally handles CONNECT tunneling
- **Verified**: 2026-04-10

## Collection Layer (Original)

### ⚠️ ByteDance API cannot go through proxy
- **Symptom**: `POST jobs.bytedance.com/api/v1/search/job/posts` through proxy returns 405 Method Not Allowed
- **Root cause**: Proxy rewrites POST method, ByteDance server rejects it
- **Fix**: ByteDance API direct connection (`proxies={}` or `proxies=None`)
- **Verified**: 2026-04-10

### ⚠️ Tencent API also doesn't need proxy
- **Symptom**: Through proxy, data is incomplete (paging anomaly)
- **Fix**: Tencent `careers.tencent.com` API direct connection
- **Verified**: 2026-04-10

### ⚠️ Alibaba IP blocking 403 (exit IP flagged)
- **Symptom**: `talent.alibaba.com/position/search` and `careers.aliyun.com` both return 403
- **Root cause**: Exit IP flagged by Alibaba's risk control
- **Fallback**: agent-browser (Playwright rendering) — slower but bypasses risk control
- **Note**: Both direct and proxied connections return 403; must use browser
- **Verified**: 2026-04-10

### ⚠️ Feishu recruitment links (jobs.feishu.cn) return 404 from certain IPs
- **Symptom**: `zhipu-ai.jobs.feishu.cn`, `vrfi1sk8a0.jobs.feishu.cn`, `moonshot.jobs.feishu.cn` all 404
- **Proxy also fails**: Some proxy IPs are also blocked by Feishu
- **Fallback**: Collect from official sites (zhipuai.cn, minimax.io, moonshot.cn) or mokahr.com
- **Verified**: 2026-04-10

### ⚠️ ByteDance job location field is often empty
- **Symptom**: `city_info_list` exists but `cn_name` is empty string
- **Cause**: Some ByteDance jobs are posted across multiple cities, location field left blank
- **Handling**: Empty location displayed as "—", doesn't affect total counts
- **Verified**: 2026-04-10

### ⚠️ cron --tz has no effect on --at, must manually convert to UTC
- **Symptom**: `--tz Asia/Shanghai --at "18:00"` actually runs at UTC 18:00 (= Beijing 02:00)
- **Fix**: Beijing 18:00 = UTC 10:00, write `0 10 * * *` directly (no --tz)
- **Verified**: 2026-04-09

## Push Layer

### ⚠️ Document push may timeout with very long content
- **Symptom**: Reports over 50KB may timeout (60s)
- **Handling**: Truncate very long reports to 50000 chars before push; save full text locally as report.md
- **Status**: Not triggered, monitoring

### ⚠️ Parent document ID must be a numeric string
- **Symptom**: Passing `"null"` or `""` causes errors
- **Fix**: `--parentId` parameter only passed when it has a value
- **Verified**: 2026-04-09

## Data Quality

### ⚠️ Tencent job count fluctuates significantly (25→47→103→135)
- **Cause**: Different API parameters (with/without area=cn, different pageSize) yield different result sets
- **Current config**: `area=cn, pageSize=50, keyword loop` — results are relatively stable
- **Note**: Tencent job dedup uses `id: make_job_id()` as canonical count
- **Verified**: 2026-04-10

### ⚠️ L1 keyword "agent" (lowercase) matches non-Agent positions
- **Symptom**: "Software Engineer - Sales Agent" gets matched
- **Current handling**: Accept this noise — precision ~90%+
- **Future**: Can add title post-processing filter rules
- **Verified**: 2026-04-09

## Cron / Deployment

### ⚠️ run_daily.sh uses `set -e`, but collection failure stops subsequent scripts
- **Symptom**: If daily_collect.py exits non-zero, diff and push won't run
- **Handling**: daily_collect.py internally catches all company exceptions, main function returns 0
- **Verified**: Confirmed 403/Feishu 404 exceptions caught internally, not thrown to outer scope
- **Verified**: 2026-04-10

### ⚠️ cron delivery.mode=none causes no execution notification
- **Fixed**: Payload message changed to use message tool for IM notification after execution
- **Fixed**: 2026-04-10
